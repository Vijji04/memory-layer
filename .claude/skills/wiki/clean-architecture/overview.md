## Concept Index (quick nav)

All concepts in this wiki:

- [[Abstract-Factory-Pattern-A-creational-pattern-that-isolates-the-source-code-depe]] — Abstract Factory Pattern: A creational pattern that isolates the source code dependency on a concrete object's constructor behind an abstract factory interface, keeping the abstract side of the boundary clean
- [[Abstract-Use-Case-Inheritance-Represent-shared-behavior-between-similar-use-case]] — Abstract Use Case Inheritance: Represent shared behavior between similar use cases as an abstract use case, implemented via abstract classes in a shared component that concrete use cases inherit from
- [[Actor-Driven-Partitioning-Identify-the-distinct-actors-sources-of-change-first-t]] — Actor-Driven Partitioning: Identify the distinct actors (sources of change) first, then partition system components so that changes to one actor do not affect any other actor
- [[Actor-based-Cohesion-Cohesion-is-defined-by-grouping-code-that-serves-the-same-a]] — Actor-based Cohesion: Cohesion is defined by grouping code that serves the same actor, not merely code that 'does one thing' functionally
- [[Acyclic-Dependencies-Principle-ADP-The-component-dependency-graph-must-be-a-dire]] — Acyclic Dependencies Principle (ADP): The component dependency graph must be a directed acyclic graph (DAG); no cycles allowed, or all components in the cycle collapse into one unreleasable unit
- [[App-titude-Test-Kent-Becks-three-stage-model-make-it-work-make-it-right-make-it-]] — App-titude Test: Kent Beck's three-stage model (make it work, make it right, make it fast) used to diagnose why embedded code stagnates at 'make it work' and never achieves long-term structural health
- [[Architectural-Boundary-Curved-Line-A-structural-divide-separating-the-abstract-c]] — Architectural Boundary (Curved Line): A structural divide separating the abstract component (high-level business rules) from the concrete component (implementation details), with all source code dependencies pointing toward the abstract side
- [[Architectural-Boundary-as-Testability-Boundary-Every-major-architectural-boundar]] — Architectural Boundary as Testability Boundary: Every major architectural boundary naturally separates hard-to-test infrastructure from easy-to-test logic, and the Humble Object pattern is the mechanism that enforces this separation
- [[Architectural-ISP-The-dependency-on-excess-baggage-problem-scales-beyond-code-to]] — Architectural ISP: The dependency-on-excess-baggage problem scales beyond code to system and framework dependencies, causing cascading redeployment and failure risk
- [[Architecture-Archaeology-The-practice-of-excavating-principles-of-good-architect]] — Architecture Archaeology: The practice of excavating principles of good architecture by examining real historical systems, tracing how constraints shaped structural decisions over decades
- [[Architecture-Reveals-Operation-Principle-Good-architecture-elevates-use-cases-fe]] — Architecture-Reveals-Operation Principle: Good architecture elevates use cases, features, and required behaviors to first-class visible landmarks so operational intent is self-evident in the codebase
- [[Asymmetric-Marriage-Model-The-relationship-between-developer-and-framework-is-on]] — Asymmetric Marriage Model: The relationship between developer and framework is one-sided — you commit fully to the framework, but the framework author commits nothing to you
- [[Axes-of-Change-Boundary-Detection-Identify-architectural-boundaries-by-finding-i]] — Axes-of-Change Boundary Detection: Identify architectural boundaries by finding independent dimensions along which the system might vary (e.g., language vs. communication mechanism vs. storage type)
- [[Axis-of-Change-The-architectural-level-analog-of-SRP-architectural-boundaries-ar]] — Axis of Change: The architectural-level analog of SRP — architectural boundaries are drawn along the lines of actors whose change requests must be isolated from each other
- [[Behavior-vs-Architecture-Duality-Software-has-two-values-behavior-what-it-does-n]] — Behavior vs. Architecture Duality: Software has two values — behavior (what it does now) and architecture (how easy it is to change); both must be maintained, but architecture is the greater long-term value
- [[Boundary-Lines-Architectural-separations-drawn-between-components-that-change-at]] — Boundary Lines: Architectural separations drawn between components that change at different rates and for different reasons, preventing change propagation across the boundary
- [[Böhm-Jacopini-Theorem-All-programs-can-be-constructed-from-exactly-three-control]] — Böhm-Jacopini Theorem: All programs can be constructed from exactly three control structures — sequence, selection, and iteration — and nothing more is needed
- [[CCP-Common-Closure-Principle-Classes-that-change-for-the-same-reasons-and-at-the]] — CCP (Common Closure Principle): Classes that change for the same reasons and at the same times belong in the same component; classes that change for different reasons belong in different components
- [[CRP-Common-Reuse-Principle-Classes-and-modules-that-are-reused-together-belong-t]] — CRP (Common Reuse Principle): Classes and modules that are reused together belong together; classes not tightly coupled should not share a component, to avoid forcing unnecessary redeployment
- [[Central-Transform-Pattern-The-component-farthest-from-all-inputs-and-outputs-is-]] — Central Transform Pattern: The component farthest from all inputs and outputs is the highest-level, most stable element (Meilir Page-Jones's 'Central Transform')
- [[Clean-Architecture-A-concentric-layer-system-architecture-where-source-code-depe]] — Clean Architecture: A concentric-layer system architecture where source code dependencies point only inward, separating entities, use cases, interface adapters, and frameworks/drivers
- [[Common-Closure-Principle-The-component-level-analog-of-SRP-components-should-be-]] — Common Closure Principle: The component-level analog of SRP — components should be closed to changes from the same set of actors
- [[Component-Cohesion-Tension-Triangle-A-three-vertex-tension-diagram-showing-REP-C]] — Component Cohesion Tension Triangle: A three-vertex tension diagram showing REP, CCP, and CRP as opposing forces — REP and CCP are inclusive (make components larger), CRP is exclusive (makes components smaller), and good architecture navigates the balance
- [[Component-Plugin-Architecture-Dynamically-linked-files-jar-dll-shared-libraries-]] — Component Plugin Architecture: Dynamically linked files (.jar, .dll, shared libraries) that can be composed at runtime, making independent deployability the default rather than a heroic effort
- [[Component-as-Deployment-Unit-Components-are-the-smallest-independently-deployabl]] — Component as Deployment Unit: Components are the smallest independently deployable entities in a system, mapping to jar/gem/DLL/shared-library artifacts depending on platform
- [[Component-Based-Services-Services-designed-with-internal-SOLID-compliant-compone]] — Component-Based Services: Services designed with internal SOLID-compliant component architectures allow new features to be added via new components (jar files, DLLs) without modifying existing service code, conforming to the Open-Closed Principle
- [[Configuration-Driven-Dispatch-When-implementations-are-not-substitutable-isolate]] — Configuration-Driven Dispatch: When implementations are not substitutable, isolate the variation into a configuration database keyed by identifier rather than embedding conditionals in business logic
- [[Conways-Law-Applied-System-architecture-mirrors-team-communication-structure-the]] — Conway's Law Applied: System architecture mirrors team communication structure; therefore, component isolation must be designed to enable independent team action
- [[Critical-Business-Data-The-data-that-the-Critical-Business-Rules-operate-on-whic]] — Critical Business Data: The data that the Critical Business Rules operate on, which would exist even without an automated system
- [[Critical-Business-Rules-Rules-that-make-or-save-the-business-money-irrespective-]] — Critical Business Rules: Rules that make or save the business money irrespective of automation — they would exist even if executed manually
- [[Cross-Cutting-Concern-Vulnerability-Functionally-decomposed-service-graphs-are-i]] — Cross-Cutting Concern Vulnerability: Functionally decomposed service graphs are inherently fragile when new features must touch every service; OCP-compliant polymorphic component design is the countermeasure
- [[D-Metric-Distance-from-Main-Sequence-D-A-I---1-measuring-how-far-a-component-dev]] — D Metric (Distance from Main Sequence): D = |A + I - 1|, measuring how far a component deviates from the Main Sequence; values near 0 are ideal, values near 1 indicate architectural problems
- [[Data-Model-vs-Data-Storage-Separation-The-structure-given-to-data-within-the-app]] — Data-Model vs. Data-Storage Separation: The structure given to data within the application (data model) is architecturally significant, but the software system used to persist it (database) is not
- [[Database-Gateway-Interface-A-polymorphic-interface-in-the-use-case-layer-whose-i]] — Database Gateway Interface: A polymorphic interface in the use-case layer whose implementation (SQL/ORM) lives in the database layer as a humble object, enabling interactors to be tested with stubs
- [[Database-as-Detail-The-database-is-a-low-level-mechanism-like-a-doorknob-that-do]] — Database-as-Detail: The database is a low-level mechanism (like a doorknob) that does not rise to architectural significance; only the data model is architecturally relevant
- [[Decoupled-Layers-Horizontal-partitioning-of-the-system-into-UI-application-speci]] — Decoupled Layers: Horizontal partitioning of the system into UI, application-specific business rules, application-independent business rules, and database so each can change independently
- [[Decoupled-Use-Cases-Vertical-slicing-of-the-system-so-each-use-case-cuts-through]] — Decoupled Use Cases: Vertical slicing of the system so each use case cuts through all horizontal layers independently, allowing addition of new use cases without disturbing old ones
- [[Decoupling-Fallacy-Services-are-not-strongly-decoupled-merely-because-they-run-i]] — Decoupling Fallacy: Services are not strongly decoupled merely because they run in separate processes; they remain coupled through shared data schemas and behavioral contracts
- [[Decoupling-Mode-Spectrum-Three-levels-of-decouplingsource-level-deployment-level]] — Decoupling Mode Spectrum: Three levels of decoupling—source-level, deployment-level, and service-level—representing a progression that a good architecture can traverse in either direction as needs change
- [[Defer-Decisions-Framework-Shape-the-system-so-that-the-choice-of-database-web-fr]] — Defer Decisions Framework: Shape the system so that the choice of database, web framework, deployment mechanism, and other details can be postponed until maximum information is available
- [[Deferred-Decision-Pattern-Placing-interfaces-between-business-logic-and-infrastr]] — Deferred Decision Pattern: Placing interfaces between business logic and infrastructure details so that concrete technology choices can be delayed until sufficient information exists to make them wisely
- [[Deferred-Framework-Commitment-Treat-frameworks-databases-web-servers-and-deliver]] — Deferred Framework Commitment: Treat frameworks, databases, web servers, and delivery mechanisms as options to be decided later, keeping them decoupled from core architecture
- [[Delivery-Mechanism-Agnosticism-Design-the-system-so-it-could-be-delivered-as-a-c]] — Delivery-Mechanism Agnosticism: Design the system so it could be delivered as a console app, web app, thick client, or web service without fundamental architectural change
- [[Dependency-Arrow-Rule-All-dependency-arrows-between-components-must-point-toward]] — Dependency Arrow Rule: All dependency arrows between components must point toward core business rules; lower-level details depend on higher-level abstractions, never the reverse
- [[Dependency-Injection-Boundary-All-DI-framework-usage-is-confined-to-Main-depende]] — Dependency Injection Boundary: All DI framework usage is confined to Main; dependencies are then distributed normally (without the framework) throughout the rest of the system
- [[Dependency-Inversion-Principle-DIP-Source-code-dependencies-should-refer-only-to]] — Dependency Inversion Principle (DIP): Source code dependencies should refer only to abstractions, not concretions, so that volatile implementations can change without affecting high-level policy
- [[Dependency-Inversion-Principle-applied-at-boundaries-Using-interfaces-and-polymo]] — Dependency Inversion Principle (applied at boundaries): Using interfaces and polymorphism to make source code dependencies oppose the flow of control, enabling inner layers to call outer layer behavior without naming outer layer types
- [[Dependency-Inversion-via-Dynamic-Polymorphism-Using-interfaces-and-runtime-dispa]] — Dependency Inversion via Dynamic Polymorphism: Using interfaces and runtime dispatch to invert compile-time dependencies against the flow of control so higher-level components remain independent of lower-level details
- [[Dependency-Inversion-Using-polymorphism-interfacesabstract-types-to-reverse-the-]] — Dependency Inversion: Using polymorphism (interfaces/abstract types) to reverse the direction of source code dependencies so that high-level policy modules do not depend on low-level detail modules
- [[Dependency-Rule-Applied-to-Frameworks-Source-code-dependencies-must-point-inward]] — Dependency Rule Applied to Frameworks: Source code dependencies must point inward; frameworks must not be imported into inner architectural circles
- [[Dependency-Rule-Orientation-All-source-code-dependencies-point-toward-higher-lev]] — Dependency Rule Orientation: All source code dependencies point toward higher-level policy components; arrows point up in architectural diagrams toward the component with the highest-level policies
- [[Dependency-Rule-in-Component-Graphs-Ensure-all-cross-boundary-dependencies-point]] — Dependency Rule in Component Graphs: Ensure all cross-boundary dependencies point toward higher-level policy components; use OCP (open arrows with flow, closed inheritance arrows against flow) to enforce this
- [[Dependency-Rule-The-overriding-rule-that-source-code-dependencies-must-point-onl]] — Dependency Rule: The overriding rule that source code dependencies must point only inward, toward higher-level policies, so nothing in an inner circle can know anything about an outer circle
- [[Dependency-Inverted-Boundary-A-boundary-where-the-flow-of-control-runs-opposite-]] — Dependency-Inverted Boundary: A boundary where the flow of control runs opposite to compile-time dependencies — achieved via a polymorphic interface so the supervisor can invoke applications without knowing anything about them
- [[Dependency-Normal-Boundary-A-boundary-where-compile-time-dependencies-align-with]] — Dependency-Normal Boundary: A boundary where compile-time dependencies align with the flow of control — applications depend on the supervisor, and control flows from applications to supervisor
- [[Deployment-Level-Decoupling-Mode-Boundaries-represented-as-dynamically-linked-li]] — Deployment-Level Decoupling Mode: Boundaries represented as dynamically linked libraries (DLLs, JARs, Gems), delivered in binary form without recompilation
- [[Design-for-Testability-Principle-The-discipline-of-treating-tests-as-first-class]] — Design for Testability Principle: The discipline of treating tests as first-class architectural components whose needs shape the design of the production system, not an afterthought
- [[Design-Architecture-Continuum-High-level-architecture-and-low-level-design-detai]] — Design-Architecture Continuum: High-level architecture and low-level design details are not separate concerns but form a single continuous fabric; separating them is a false dichotomy
- [[Design-Architecture-Continuum-The-assertion-that-high-level-architecture-and-low]] — Design-Architecture Continuum: The assertion that high-level architecture and low-level design decisions are not categorically different — they exist on a single continuous spectrum of decisions that all serve the same goal of defining system structure
- [[Directed-Acyclic-Graph-of-Policy-Components-Components-containing-same-level-pol]] — Directed Acyclic Graph of Policy Components: Components containing same-level policies form nodes; compile-time dependencies form directed edges always pointing from lower-level to higher-level components
- [[Directional-Dependency-Control-Using-interfaces-and-abstractions-to-invert-depen]] — Directional Dependency Control: Using interfaces and abstractions to invert dependency arrows so they point toward the components that must be protected from change
- [[DualMulti-Stream-Data-Flow-Complex-systems-decompose-into-multiple-independent-d]] — Dual/Multi-Stream Data Flow: Complex systems decompose into multiple independent data streams (e.g., user communication stream vs. persistence stream) that converge at the highest-level policy component
- [[Eisenhower-Urgency-Importance-Matrix-A-2x2-grid-sorting-tasks-by-urgency-and-imp]] — Eisenhower Urgency-Importance Matrix: A 2x2 grid sorting tasks by urgency and importance, used to argue that architecture (important, not urgent) must not be sacrificed to features (urgent, not always important)
- [[Entity-A-software-object-that-binds-Critical-Business-Rules-and-Critical-Busines]] — Entity: A software object that binds Critical Business Rules and Critical Business Data together in a single, isolated module free from UI, DB, and framework concerns
- [[Euclidean-Proof-Hierarchy-Abandoned-Ideal-Dijkstras-original-vision-of-building-]] — Euclidean Proof Hierarchy (Abandoned Ideal): Dijkstra's original vision of building software correctness via formal mathematical proofs analogous to geometric theorems — valuable as motivation but impractical in full form
- [[Event-Sourcing-Store-only-transactions-CR-never-mutable-state-so-current-state-i]] — Event Sourcing: Store only transactions (CR), never mutable state, so current state is always derived by replaying transactions from a known checkpoint
- [[Facade-Pattern-for-SRP-A-thin-EmployeeFacade-class-instantiates-and-delegates-to]] — Facade Pattern for SRP: A thin EmployeeFacade class instantiates and delegates to separate actor-aligned classes, reducing cognitive overhead while maintaining separation
- [[Facade-Pattern-Define-a-boundary-via-a-single-Facade-class-that-proxies-all-serv]] — Facade Pattern: Define a boundary via a single Facade class that proxies all service calls, sacrificing dependency inversion for simplicity
- [[Firmware-vs-Software-Distinction-Code-is-firmware-not-because-of-where-it-is-sto]] — Firmware vs. Software Distinction: Code is 'firmware' not because of where it is stored but because of how tightly it depends on hardware—unmanaged hardware dependencies make any code firmware regardless of storage medium
- [[Flexible-Deployment-Grouping-Structure-the-compilebuild-environment-at-fine-gran]] — Flexible Deployment Grouping: Structure the compile/build environment at fine granularity (per-actor, per-layer components) while reserving the right to combine deliverables into coarser groups at deploy time

- [[Functional-Decomposition-The-practice-of-breaking-large-problem-statements-into-]] — Functional Decomposition: The practice of breaking large problem statements into high-level functions, then recursively decomposing those into smaller provable units, enabled by structured programming
- [[Functional-Programming-Imposes-discipline-upon-assignment-by-treating-values-as-]] — Functional Programming: Imposes discipline upon assignment by treating values as immutable, eliminating or strictly limiting assignment statements
- [[Hardware-Abstraction-Layer-HAL-A-boundary-layer-that-isolates-software-from-hard]] — Hardware Abstraction Layer (HAL): A boundary layer that isolates software from hardware details, allowing hardware to change without propagating changes into business logic or domain code
- [[Hexagonal-Architecture-Ports-and-Adapters-An-architecture-by-Alistair-Cockburn-t]] — Hexagonal Architecture (Ports and Adapters): An architecture by Alistair Cockburn that separates concerns by defining explicit ports for external interaction and adapters that implement them
- [[Humble-Object-Pattern-Split-behaviors-into-two-modules-one-humble-hard-to-test-s]] — Humble Object Pattern: Split behaviors into two modules — one humble (hard-to-test, stripped to bare essence) and one testable (all the logic that was extracted from the humble object)
- [[Immutability-Principle-Variables-in-functional-languages-do-not-vary-eliminating]] — Immutability Principle: Variables in functional languages do not vary, eliminating entire classes of concurrency bugs by design
- [[Independent-DeployabilityDevelopability-When-source-code-dependencies-are-invert]] — Independent Deployability/Developability: When source code dependencies are inverted so components depend only on abstractions, each component can be compiled, deployed, and developed independently
- [[Independent-Development-Fallacy-Services-cannot-always-be-independently-develope]] — Independent Development Fallacy: Services cannot always be independently developed and deployed because data or behavioral coupling forces coordinated changes across service boundaries
- [[Inflection-Point-Boundary-Investment-Delay-full-boundary-implementation-until-th]] — Inflection Point Boundary Investment: Delay full boundary implementation until the cost of implementing becomes less than the cost of ignoring, monitored continuously as the system evolves
- [[Information-Hiding-via-Interface-Segregation-Introducing-interfaces-not-just-to-]] — Information Hiding via Interface Segregation: Introducing interfaces not just to invert dependencies but to prevent transitive dependencies from leaking internals of one component into another
- [[Interface-Adapters-Layer-The-layer-that-converts-data-between-the-format-conveni]] — Interface Adapters Layer: The layer that converts data between the format convenient for use cases/entities and the format required by external agencies like databases or UIs
- [[Interface-Segregation-Principle-ISP-Users-should-not-be-forced-to-depend-on-inte]] — Interface Segregation Principle (ISP): Users should not be forced to depend on interfaces they do not use; split fat interfaces into smaller, role-specific ones
- [[Life-Cycle-Support-Model-Evaluate-architecture-not-by-whether-the-system-runs-bu]] — Life-Cycle Support Model: Evaluate architecture not by whether the system runs, but by how well it supports development, deployment, operation, and maintenance over its full lifetime
- [[Linking-Loader-Evolution-The-historical-split-of-linking-slow-done-ahead-of-time]] — Linking Loader Evolution: The historical split of linking (slow, done ahead-of-time) from loading (fast, done at runtime) that ultimately enabled modern plugin/component architectures
- [[Liskov-Substitution-Principle-LSP-If-S-is-a-subtype-of-T-then-objects-of-type-S-]] — Liskov Substitution Principle (LSP): If S is a subtype of T, then objects of type S must be substitutable for objects of type T without altering the correctness of the program
- [[Local-Process-Boundary-Separate-address-spaces-on-the-same-processor-communicati]] — Local Process Boundary: Separate address spaces on the same processor, communicating via OS facilities (sockets, mailboxes, message queues); moderately expensive
- [[Main-Sequence-The-ideal-locus-of-component-positions-on-the-Abstractness-Instabi]] — Main Sequence: The ideal locus of component positions on the Abstractness-Instability graph, connecting (A=1, I=0) and (A=0, I=1), representing the optimal balance between abstraction and stability
- [[Main-as-Plugin-Main-is-treated-as-a-replaceable-plugin-to-the-application-not-as]] — Main-as-Plugin: Main is treated as a replaceable plugin to the application, not as the application itself — it sets up conditions and hands control upward
- [[Minimum-Effort-Architecture-Goal-Good-software-architecture-is-defined-solely-by]] — Minimum-Effort Architecture Goal: Good software architecture is defined solely by its ability to minimize the human resources required to build and maintain a system over its lifetime
- [[Object-Oriented-Programming-Imposes-discipline-on-indirect-transfer-of-control-b]] — Object-Oriented Programming: Imposes discipline on indirect transfer of control by enabling polymorphism through disciplined use of function pointers abstracted into classes and methods
- [[One-Dimensional-Boundary-Strategy-Pattern-Use-a-single-interface-to-invert-depen]] — One-Dimensional Boundary (Strategy Pattern): Use a single interface to invert dependency in one direction, setting the stage for a full boundary without the cost of reciprocal isolation
- [[Open-Closed-Principle-OCP-A-software-artifact-should-be-open-for-extension-but-c]] — Open-Closed Principle (OCP): A software artifact should be open for extension but closed for modification — behavior should be extendible without altering the artifact itself

- [[Options-Open-Strategy-The-primary-architectural-value-is-keeping-as-many-structu]] — Options-Open Strategy: The primary architectural value is keeping as many structural options open as long as possible, preserving the 'softness' of software
- [[Overlay-System-Pattern-A-memory-management-architecture-where-applications-are-l]] — Overlay System Pattern: A memory management architecture where applications are loaded from disk into a fixed overlay area, executed until buffers fill, then swapped out to allow other programs to run, enabling many programs to coexist in limited RAM
- [[Package-by-Component-Hybrid-approach-bundling-all-responsibilities-of-a-coarse-g]] — Package by Component: Hybrid approach bundling all responsibilities of a coarse-grained component into a single package, using access modifiers to let the compiler enforce architectural rules
- [[Package-by-Feature-Vertical-slicing-of-code-by-domain-concept-or-feature-top-lev]] — Package by Feature: Vertical slicing of code by domain concept or feature; top-level structure screams business domain but still suboptimal for enforcing boundaries
- [[Package-by-Layer-Horizontal-slicing-of-code-into-technical-layers-web-service-pe]] — Package by Layer: Horizontal slicing of code into technical layers (web, service, persistence); simple to start but doesn't reflect business domain and enables architectural bypass
- [[Paradigms-as-Constraints-Model-A-meta-framework-recognizing-that-programming-par]] — Paradigms-as-Constraints Model: A meta-framework recognizing that programming paradigms define progress by removing dangerous capabilities rather than adding new ones
- [[Performance-Encapsulation-Principle-Performance-concerns-around-data-storage-are]] — Performance-Encapsulation Principle: Performance concerns around data storage are real but must be encapsulated at the low-level boundary, not allowed to shape overall architecture
- [[Plugin-Architecture-for-Low-Level-Components-Lower-level-components-are-designed]] — Plugin Architecture for Low-Level Components: Lower-level components are designed as plugins to higher-level ones, so the high-level component knows nothing of the low-level implementation
- [[Plugin-Architecture-A-design-pattern-where-low-level-detail-modules-UI-database-]] — Plugin Architecture: A design pattern where low-level detail modules (UI, database, devices) are plugged into high-level policy modules, making high-level code unaware of low-level implementations
- [[Plugin-Architecture-Core-business-rules-are-kept-independent-of-optional-or-vari]] — Plugin Architecture: Core business rules are kept independent of optional or variable components (GUI, database, frameworks) which plug into the core rather than being coupled to it
- [[Policy-Level-Hierarchy-Policies-are-ranked-by-their-distance-from-system-inputs-]] — Policy-Level Hierarchy: Policies are ranked by their distance from system inputs and outputs; higher distance equals higher level, and source code dependencies must point toward higher-level policies
- [[Policy-vs-Details-Separation-Decompose-every-system-into-high-level-policy-busin]] — Policy-vs-Details Separation: Decompose every system into high-level policy (business rules) and low-level details (IO, DB, frameworks, protocols); architect so policy has zero knowledge of details
- [[Polymorphism-as-Controlled-Indirection-OOs-core-contribution-is-making-function-]] — Polymorphism as Controlled Indirection: OO's core contribution is making function-pointer-based dispatch safe and implicit, giving architects absolute control over source code dependency direction
- [[Ports-and-Adapters-Hexagonal-Architecture-Code-organized-into-inside-domain-and-]] — Ports and Adapters (Hexagonal Architecture): Code organized into 'inside' (domain) and 'outside' (infrastructure) regions where outside always depends on inside, never the reverse
- [[PresenterView-Split-The-View-is-the-humble-object-that-only-moves-pre-formatted-]] — Presenter/View Split: The View is the humble object that only moves pre-formatted data to the screen; the Presenter is the testable object that formats all data into a View Model of strings, booleans, and enums
- [[Productivity-Asymptote-Model-When-code-quality-is-neglected-developer-output-app]] — Productivity Asymptote Model: When code quality is neglected, developer output approaches zero asymptotically over successive releases even as headcount and cost grow, producing a predictable 'signature of a mess'
- [[Protection-Hierarchy-A-component-dependency-structure-where-higher-level-more-ce]] — Protection Hierarchy: A component dependency structure where higher-level (more central) components are shielded from change by having lower-level components depend on them, not the reverse
- [[Périphérique-Anti-Pattern-Placing-all-infrastructure-code-in-one-source-tree-all]] — Périphérique Anti-Pattern: Placing all infrastructure code in one source tree allows infrastructure components to call each other directly, bypassing the domain entirely
